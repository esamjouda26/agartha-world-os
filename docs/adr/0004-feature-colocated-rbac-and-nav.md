# 0004 — Feature-Colocated RBAC + Nav

**Status:** Accepted — 2026-04-20
**Decision owner:** Phase 4 / pre-Phase-5 refactor
**Supersedes:** the three hand-authored central files — `src/lib/rbac/sidebar-config.ts`, `src/lib/rbac/navigation.ts`, `src/lib/rbac/route-manifest.ts` (all deleted).

---

## Context

Mid-Phase-4 the system carried four overlapping sources encoding
"who-sees/enters-what":

1. `frontend_spec.md §8` — the specification.
2. `src/lib/rbac/sidebar-config.ts` (536 LOC) — hand-authored sidebar entries with visibility rules.
3. `src/lib/rbac/route-manifest.ts` — hand-authored middleware Gate-5 table.
4. Migration RLS policies — the actual database enforcement.

A concrete drift was found: `frontend_spec.md §8c` gates `/crew/incidents`
on `ops:c`, `route-manifest.ts` repeats this, but the `incidents_insert`
RLS policy in the init migration accepts any authenticated user. F&B
crew couldn't see the nav item, even though the DB would accept their
INSERT. Four-way drift is ongoing — only the specific incident was
triaged.

Per CLAUDE.md §1 (DDD colocation), §4 (Data Protection Parity —
"frontend visual RBAC is UX only; MUST be backed by RLS"), §9 (Proactive
Architectural Optimization), and §10 (Zero Projection — schema is the
sole source of truth), the hand-authored central files violate house
standards.

Plans A and B (audited separately) both proposed feature-colocated
manifests. Plan A coupled UI visibility and route security in one
structure — reintroducing the original drift class for deep-link routes.
Plan B split concerns but used regex (ReDoS risk on the Edge), bundled
nav metadata into the middleware graph (Edge-bundle bloat), and did not
address the UI ↔ RLS drift.

## Decision

Adopt **Plan C**:

### Per-feature colocation

Each `src/features/<feature>/` owns two sibling files:

- **`rbac.ts`** — security only. Exports `FeatureRbac = { featureId, routes: RouteRequirement[] }`. Entries carry `pattern` (URLPattern syntax), `domain`, `access`, `primaryTables` (RLS-parity reference).
- **`nav.ts`** — UI only. Exports `FeatureNav = { featureId, items: FeatureNavItem[] }`. Entries carry `portal`, `path`, `section`, `order`, `iconName`, `labelKey`, `label`, optional `requires` / `excludeWhenAccessLevel` / `excludeWhenHoldsAnyDomain` filters.

Deep-link-only routes (e.g. `/admin/iam/:id`) live in `rbac.ts` only —
never in `nav.ts`. Shared staff routes (Attendance, Settings, etc.) live
in `nav.ts` only — middleware lets them through via the bypass allowlist
and RLS enforces row-level access.

### Two disjoint aggregators

- `src/lib/rbac/middleware-manifest.ts` — imports only `*/rbac.ts`. Compiles URLPatterns once at module load, sorts most-specific-first.
- `src/lib/nav/manifest.ts` + `src/lib/nav/filter.ts` — import only `*/nav.ts` + `src/lib/nav/sections.ts`. Portal layouts and the command palette call `filterNavForUser(portal, accessLevel, domains)`.

Middleware imports only from `src/lib/rbac/**`; never from
`src/lib/nav/**`. ESLint `no-restricted-imports` enforces the firewall.

### URLPattern, not regex

Feature `rbac.ts` files use `URLPattern` pathname syntax (native in Node
23+, Vercel Edge Runtime, modern browsers). Regex literals are banned
inside `src/features/*/rbac.ts` by ESLint `no-restricted-syntax`. This
eliminates ReDoS risk and avoids per-request RegExp compilation.
Cross-runtime determinism is verified by `scripts/verify-urlpattern-runtime.ts`.

### Specificity ordering

Patterns are sorted at module load by `specificityScore` — literal-char
count × 1000 + named-param-count × 10 − wildcard-count × 100. More
literal scaffolding wins. Deterministic, documented, covered by unit
tests in `src/lib/rbac/__tests__/policy.test.ts`.

### Locale-stripped branded type

`LocaleStrippedPath` is a branded string produced only by
`brandLocaleStripped`, which is called exactly once inside
`middleware.ts#stripLocale`. All policy functions accept only this
branded type — tsc prevents accidental locale-prefixed lookups.
ESLint rule bans `as LocaleStrippedPath` casts outside middleware.

### Generated `DomainCode`

`src/lib/rbac/domains.generated.ts` is regenerated by
`scripts/generate-domain-codes.ts` from the
`INSERT INTO permission_domains (code, ...)` block in
`supabase/migrations/20260417064731_init_schema.sql`. `rbac.ts` authors
use `domain: <literal>` which is typechecked against the generated
union — typos fail tsc.

### CI gates

1. `rbac:pattern-uniqueness` — no two features declare the same pattern.
2. `rbac:orphan-routes` — every staff `page.tsx` is either pattern-matched by some `rbac.ts` OR in `SHARED_BYPASS_PREFIXES`.
3. `rbac:rls-parity` — for every route with non-empty `primaryTables`, asserts the feature's tables have `CREATE POLICY` clauses whose `USING` references `domains->'<domain>' ? '<access>'` for the corresponding SQL command (c→INSERT, r→SELECT, u→UPDATE, d→DELETE).
4. `rbac:bundle-budget` — `.next/server/middleware.js` must stay below 800 KB (80% of Vercel's 1 MB Edge-middleware cap).

### No holding pen — single source of truth

An earlier draft of this ADR proposed a transitional
`src/features/_pending/` folder to keep middleware Gate-5 coverage
intact for routes whose feature folders don't exist yet. That holding
pen was **removed** before the first cleanup commit because it
reintroduced exactly the drift class this refactor closes: a second
declarative surface listing routes that don't correspond to real
features, which ran ahead of the sidebar's true implementation state.

After the removal:

- A route exists in the sidebar **only** when the feature owning it
  lands a `nav.ts`.
- A route is Gate-5 gated **only** when the feature owning it lands a
  `rbac.ts`.
- An unimplemented staff route that no feature has claimed and no
  bypass covers is caught by the `rbac:orphan-routes` CI gate — the
  author must either add it to the feature's rbac.ts or add the prefix
  to `SHARED_BYPASS_PREFIXES` (for true cross-portal shared routes and
  portal landings like `/admin/it`, `/admin/business`, `/management`).

Portal landing pages (`/admin/it`, `/admin/business`, `/management`)
are Gate-4-gated (portal ↔ access_level) and appear in
`SHARED_BYPASS_PREFIXES`; they are not domain-gated because they render
only the user's own profile display name with no cross-user data.

## STRIDE — security threat model

| Threat                     | Vector                                            | Mitigation                                                                                                                                                                                   |
| -------------------------- | ------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **S**poofing               | Forged `domains` claim on JWT                     | Supabase signs JWT; claim is trusted server-side only. Unchanged from pre-refactor.                                                                                                          |
| **T**ampering              | Malicious edit to `rbac.ts` to widen access       | CODEOWNERS requires 2 security-team approvers on `src/features/*/rbac.ts` and `src/lib/rbac/**`. Signed commits on `main` (CLAUDE.md §13).                                                   |
| **R**epudiation            | User disputes a denied access attempt             | Audit logging on Gate 5 denials is **descoped from this refactor** (see "Known Gap" below). Failed-auth trail currently relies on Sentry + Supabase auth logs until the follow-up lands.     |
| **I**nformation Disclosure | Sidebar leaks existence of protected routes       | Nav items filtered server-side in RSC layouts; unauthorized items never serialize to the client wire. `/admin/iam/:id` style detail pages are rbac-only (never in nav.ts).                   |
| **D**enial of Service      | ReDoS via crafted URL                             | URLPattern (no backtracking) + ESLint ban on RegExp in `rbac.ts` + Vercel Edge 50 ms CPU timeout.                                                                                            |
| **E**levation of Privilege | Drift between `rbac.ts` UI claim and RLS DB claim | `rbac:rls-parity` CI gate asserts the two stay aligned for every route with declared `primaryTables`. Fully enforcing from Session 5 onwards; Phases 5–9 expand coverage feature by feature. |

## Known gaps

- **Audit logging on Gate 5 denials is descoped** from this refactor because pino is not Edge-Runtime-safe and CLAUDE.md §8 forbids `console.log` in production paths. The current plan of record: enqueue denial events from middleware to Upstash Redis, drain via pg_cron Edge Function into `system_audit_log` (append-only, 7-year retention per §11). Tracked as a follow-up; spec captured in this repo's post-refactor notes.
- **Offline PWA queue** — an earlier iteration of this ADR proposed hooking the offline queue into denial audit. The offline queue was removed from the project; this gap is closed.

## Consequences

### Positive

- UI visibility and route security mechanically decoupled; sidebar drift cannot produce auth-bypass.
- Deep-link-only routes have clean ownership: rbac.ts yes, nav.ts no.
- Middleware bundle stays lean — enforced by ESLint + CI budget gate.
- Domain-code typos caught by tsc.
- RLS ↔ UI drift detectable at CI time (the original Phase-4 bug class).
- Feature authors own both surfaces of their work in one folder; CODEOWNERS gates the security file separately.

### Negative

- Two files per feature instead of one. Justified by §11 (security review CODEOWNERS), §14 (Edge bundle budget), and CLAUDE.md §13 (signed commits / linear history — fewer files = fewer commits to audit, but the firewall is the point).
- A new staff route landing without its feature's `rbac.ts` + `nav.ts` will fail the `rbac:orphan-routes` CI gate. Authors must add the rbac entry at PR time (or mark the page as a bypass). This is the intended trade-off.
- CI adds 4 new scripts + 1 codegen script to the pipeline. Runtime negligible.

## Alternatives rejected

- **Plan A** (single-file colocation). Couples UI and security in one data structure; deep-link routes have no expression; reintroduces the drift class this ADR closes.
- **Plan B** (split inside one file). Regex patterns risk ReDoS; the single aggregator pulls nav metadata into the middleware bundle; does not close UI↔RLS drift.
- **External policy engine (OPA / Cedar / casbin)** — adds per-request network hop, violates CLAUDE.md §3 edge-speed guarantee.
- **CASL or similar policy-as-code library** — heavy, redundant with Supabase's schema-driven permission model.

## Follow-ups

- Build the Upstash-based audit-logging pipeline for Gate 5 denials (see "Known gaps").
- As each feature lands in Phases 5–9, add its `rbac.ts` + `nav.ts` with real `primaryTables`. `rbac:rls-parity` picks up coverage as tables appear; `rbac:orphan-routes` fails any PR that adds a staff `page.tsx` without a matching rbac entry or bypass.
